BUGS FOUND AND FIXED THIS SESSION
====================================================================

1. station name collision (OpenData dictionary)
   generic names like "STADE MUNICIPALE" or "MEUBLATEX" denote physically
   DIFFERENT real stops 100+ km apart in different cities. 210 of 2109
   duplicate-name entries were genuine >1km conflicts. a naive first-source-
   wins name->coord dict would silently pick the wrong location for ~10% of
   duplicated names.
   fix: keep ALL candidates per name, disambiguate using route-sequence
   geography (unique-name stops on a line anchor it; ambiguous stops take
   whichever candidate is nearest the closest anchor)

2. triangulation year-sampling bias
   triangulate_stop originally pulled tickets recent-year-first. for a
   high-volume stop (HOP.SAHLOUL: 1.78 MILLION tickets), 2025-2026 alone
   blows past the 150-ticket quota, so 2022-2024 (when S.T.S actually had
   good GPS coverage) never got sampled at all. GPS-coverage direction isn't
   universal across companies: S.R.T.K improved over time (bus 6039: zero
   pings before Nov 2025, so recent=better), but S.T.S's recent coverage is
   nearly zero while 2022-2024 was dense (recent=worse for this company)
   fix: sample a FIXED QUOTA PER YEAR, not greedy-recent-first

3. naive centroid rejection in triangulation
   original logic averaged ALL matched GPS positions and rejected the whole
   result if any point was >800m from that average. wrongly rejected a
   clean 10-of-18-point majority cluster just because 5 outlier points
   existed ~1.2km away.
   fix: find the largest internally-tight sub-cluster instead of trusting
   the global mean

4. largest-cluster is not always the correct stop (found via user ground truth)
   even after fix #3, the "largest cluster" heuristic itself is not
   reliable. for HOP.SAHLOUL, the algorithm's chosen 10-point cluster was
   confirmed WRONG by the user -- the correct location was a 4-point
   cluster (below the min_used=5 threshold!) about 1.2km away.
   root cause: S.T.S line 219 (and its other lines) is a high-frequency
   URBAN SHUTTLE that loops repeatedly through a small area. confirmed via
   one-bus-one-day ticket trace: SOUSSE -> HOP.SAHLOUL -> SOUSSE ->
   HOP.SAHLOUL -> SOUSSE, oscillating within one morning, with real
   sensible timestamps (not a batch-sync artifact). the bus almost
   certainly idles at a depot/terminal between shuttle legs (user confirmed
   STEG SAHLOUL's triangulated coord is literally the S.T.S terminal), and
   that idle point generates far more GPS pings than any brief real stop --
   creating a "gravity well" that pulls the majority-cluster triangulation
   toward it regardless of which stop name the ticket actually claims.
   13 of 15 "successful" triangulations in the top-50 test batch were S.T.S
   -- consistent with this being a company/line-pattern-specific issue, not
   random noise. of the 9 non-S.T.S candidates in that batch, 2 succeeded
   (22%) vs 13 of 41 S.T.S (32%) -- similar raw yield, the STS-heavy result
   count is because S.T.S dominates ticket volume (urban shuttles generate
   way more tickets than 1-2-trip intercity routes), not because STS has a
   uniquely different base success rate. only the CORRECTNESS is worse for
   S.T.S, confirmed by the one ground-truth check available.
   resolution: kept as real coordinates with a visible "name probably wrong"
   flag (confidence=triangule_nom_incertain) rather than deleted -- the
   physical GPS discovery is real and useful even if mislabeled.

5. GPS presence check was misleading on a 10-day window
   checking only the last 10 days for "has this company ever had GPS"
   wrongly reported S.R.T.BIZERTE, TUS, and EPE-TVE as never having GPS --
   they had substantial historical coverage (11mo, 7mo, 3wk respectively)
   that was later discontinued. a recent-window boolean can't distinguish
   "never had GPS" from "had it, then it was pulled."
   fix: sample the FULL 2022-2026 archive (every 3rd day) for historical
   first_seen/last_seen, plus a separate EXHAUSTIVE (non-sampled) scan of
   just the last 30 days for current status -- exhaustive mattered because
   S.T.S's recent activity is itself sparse (1 active day in the last 10),
   so even light sampling on the recent window could miss it (and initially
   did, until fixed)

====================================================================
MONGODB ANOMALIES -- things that are just wrong in the source DB design
====================================================================
(not data-quality noise, actual structural/schema mistakes. numbered so
they're easy to reference. re-verified against the actual DB, not from
memory.)

1. no canonical company table used as a real reference. `societe` is a
   free-text string COPIED into every collection (ligne, tickets, GPS
   pings, STOP<societe>, station, sav, details...) instead of a foreign
   key into one authoritative company table. result: the same real
   company ends up spelled differently in different places (S.R.T.K vs
   S.R.T.K0, Winicari vs winicari) with nothing in the DB enforcing or
   even detecting the drift.

2. the ONE collection that looks like it's meant to be that authoritative
   company table (winicari.societe, 10 docs, has nomComplet/Gouvernorat/
   active) is itself INCOMPLETE -- SORETRAS and S.T.C.I are entirely
   missing from it despite being real, active operators with real lines,
   tickets, and sav records elsewhere in the same database.

3. station/stop identity has SIX incompatible ID schemes for the same
   real-world physical place, none of which reliably cross-reference each
   other:
     - OpenData.*.code_station (a structured code, e.g. "42590001")
     - winicari.station.stop_id (a small integer as string)
     - STOP<societe>.NAMENRnew (an integer)
     - STOP<societe>.NAMENR (an integer, but different meaning per company
       -- see #4)
     - winicari.Names.Code (an integer, company-wide registry)
     - ticket origine/Distination codes (an integer, but PER-LINE local
       sequence, resets to 01 on every route -- NOT the same numbering as
       Names.Code even though both are small integers on the same station)
   nothing in the schema documents which of these is "the" ID, or how
   they relate. we had to empirically test each pairing one at a time to
   figure out what joins to what.

4. worse: the SAME field name (NAMENR) means DIFFERENT things depending on
   which company's STOP<societe> collection you're looking at. for
   S.R.T.BIZERTE/S.T.S/TCV, NAMENR happens to equal an OpenData
   code_station value (direct join works, 20/20 tested). for
   SRT.ELGOUAFEL/EPE-TVE, NAMENR is just a small sequential integer with
   NO relationship to OpenData at all (0/20). same field, same
   collection-family, silently different semantics per company -- would
   have produced wrong joins with false confidence if we hadn't tested
   per company before trusting it.

5. station names collide across physically distant real places under the
   EXACT same string with no dedup/validation ever enforced. "STADE
   MUNICIPALE", "MEUBLATEX" etc. -- 210 confirmed conflicts >1km apart
   among 2109 duplicate names across the OpenData collections. nothing in
   the data-entry pipeline ever checked "does this name already exist
   somewhere else" before creating a new station.

6. four parallel, overlapping OpenData station collections (Station,
   Station2, Station_new, Station_sts) that all look like sequential
   exports/migrations layered on top of each other rather than one
   collection maintained over time. entries added to newer ones were
   never merged back or used to clean up older ones -- same station can
   appear 2-4 times with slightly different coordinates depending which
   collection happened to record it.

7. `ligne.stationnames` contains at least one confirmed data-entry error
   baked directly into a route definition: S.R.T.K line 201's Tunis-side
   terminal is labeled "KASSERINE" (the city the route departs FROM, not
   the actual name of that Tunis stop). this kind of error is
   undetectable without cross-referencing an independent name source --
   trusting `ligne` alone would silently mislabel a real stop.

8. `code` (line number) is explicitly not unique across companies -- the
   same numeric code means a totally different route depending on which
   company you're looking at, with no compound key enforced anywhere
   except by convention (already known, restating here since it's a
   structural anomaly like the others).

9. no referential integrity anywhere, by construction (MongoDB has no FK
   constraints). STOP<societe>.NAMENRnew can point at a winicari.station
   stop_id or Names.Code that doesn't exist; ligne.societe can contain a
   spelling that matches none of the canonical company strings; nothing
   in the system would ever flag this. every join we built had to be
   manually verified with real hit-rate tests before trusting it, because
   nothing guarantees it holds.

10. dead/test collections left sitting in what looks like a production
    database: `stop` (7 empty docs, literally just a Java class-name
    placeholder `com.example.test11.Model.Stop`, no real fields at all),
    `historiqueStation` (mostly junk entries like `['tt','nn']`),
    `demande_ligne` (contains literal test values "aaa"/"aaaa" stored as
    real field data), `klm` (looks like a superseded/legacy predecessor to
    STOP<societe>, incomplete/inconsistent). these aren't documented as
    deprecated anywhere -- we had to sample and test each one to find out
    it was dead, and easily could have wasted time building on top of one
    before realizing.

11. GPS ping schema drifted substantially over ~4.5 years with zero
    versioning or migration markers: the `speed` field moved from nested
    `bus.vitesse` to top-level `speed` around 2025; `service.codeLigne`
    (the line link) only exists on pings from ~mid-2022 onward; ping
    density per bus jumped roughly 12x with no field marking when/why.
    anyone querying older data without already knowing this will silently
    get wrong or incomplete results.

12. GPS equipment status (does a company currently have working trackers)
    is not an explicit field anywhere -- it has to be INFERRED by scanning
    raw ping history over time, and even then a naive recent-window check
    is misleading (see companies.has_gps note above -- S.R.T.BIZERTE/TUS/
    EPE-TVE all had real GPS windows that were later silently discontinued,
    with nothing marking that transition anywhere in the data).

13. winicari.station (the coordinate table STOP<societe> depends on for
    its most reliable join) only has entries for 5 of the 12 real
    companies (S.R.T.BIZERTE, S.R.T.K, S.R.T.M, SORETRAS, Winicari) --
    an incomplete reference table for something that logically should
    cover every operator using the STOP<societe> pattern.

14. duplicate near-identical (societe, code) line documents exist inside
    `ligne` itself -- 6 found (mostly S.R.T.SELIANA), empty stub docs with
    zero stationnames, essentially dead rows with no uniqueness
    constraint stopping them from being created.

====================================================================
COMPARISON -- notebook approach (08/09/10) vs reference-DB approach
====================================================================
(figures pulled directly from the actual executed notebook outputs, not
from memory, to keep this honest)

what the notebook approach (08_station_eda / 09_station_matching /
10_foundation_before_after) actually did and got:
- scope: stop COORDINATES only. no company/line canonical tables, no
  persistent schema -- results live in notebook cell outputs plus two
  parquet caches (ticket_stop_index.parquet, a raw name->coord dict).
- method: name-match a station name against a flat OpenData dictionary
  (3358 names), with sequential geographic disambiguation as the safety
  net for name collisions, PLUS ticket-time x GPS-ping triangulation for
  names with no OpenData match at all.
- result: baseline 143 usable lines -> 187 after enrichment (+44, +31%),
  2067 -> 2383 mapped stop-rows (+316). recovery rate on the 258 dropped
  lines: 44/258 (17%).
- triangulation in the notebook phase: 0/7 on line 500's residual gaps
  (later, in THIS session's reference-db work, refined to 15/50 (30%)
  after fixing two real bugs, then rolled back to 3/15 kept after
  discovering the S.T.S "gravity well" issue -- the notebook phase never
  caught this because its triangulation sample (7 stops) was too small
  and too narrow to surface a company-specific systematic bias).
- no audit trail: nothing records WHICH source resolved a given stop, no
  confidence tiering, no way to ask "how was this coordinate obtained"
  after the fact without re-reading notebook cells.

what the reference-DB approach (this session) did and got:
- scope: full relational schema -- companies (12, enriched with the
  winicari.societe admin registry), stops (2991, built by CLUSTERING
  ~8683 raw coordinate points by physical proximity instead of matching
  by name -- solves the name-collision problem by construction, not by a
  disambiguation heuristic bolted on after the fact), lines (402, union
  of ligne/ticket/gps sources, nothing silently dropped), line_stops
  (2446 rows, 217/402 lines = 54% resolved).
- method: line_stops uses a 6-TIER resolver, direct ID joins tried FIRST
  (STOP<societe>+winicari.station, STOP<societe>.NAMENR->OpenData.code_
  station, STOP<societe>.NAMENRnew->winicari.Names->name) before falling
  back to the SAME anchor-nearest and ticket-order+disambiguation methods
  the notebooks used -- reused directly (stations.build_enriched_stops,
  stations._disambiguate_sequential), not reimplemented.
- result: of the 217 resolved lines, 160 (59/217+20/217+81/217) came from
  a DIRECT structural join needing no name-matching or probability at
  all -- these three tiers (STOP<societe>+station, NAMENR->OpenData,
  NAMENRnew->Names) were discovered THIS session and didn't exist in the
  notebook approach at all. only 57 of 217 (38+16+3) came from the
  weaker anchor/ticket-fallback tiers the notebooks already used.
- persistent, queryable, versioned: one SQLite file with real foreign
  keys, confidence tiers (verifie/inferee/non_nomme/triangule_*), notes
  columns for every caveat, and a tier label recording exactly which
  method resolved each line -- any future module reads this directly
  instead of re-deriving matches from scratch.
- bug discovery: the broader, more systematic testing this approach
  demanded (top-50 batches, per-company hit-rate checks, ground-truth
  spot-checks) surfaced 2 additional triangulation bugs (year-sampling
  bias, naive-centroid rejection) AND the S.T.S gravity-well problem --
  none of which were visible in the notebooks' narrower 0/7 test. the
  notebook approach's numbers weren't wrong, they were just too small a
  sample to expose these failure modes.

bottom line: the notebook approach was the right FIRST pass -- fast,
proved the concept, got real numbers (+44 lines) with modest effort. the
reference-DB approach is more expensive (multiple sessions, careful
per-collection verification) but produces something structurally
sounder: more of its coverage comes from verified direct joins rather
than probabilistic name-matching, it's persistent and auditable instead
of living in notebook cells, and the extra rigor it demanded is exactly
what caught bugs the lighter notebook pass missed. worth doing SEQUENTIALLY,
which is what happened -- the notebook work de-risked the concept cheaply
before committing to the heavier schema-building effort.

trips + trip_stops table -- reconstructing real GPS trips on line_stops
====================================================================

built trips/trip_stops by reusing foundation.reconstruct_bus_day (same
pipeline as build_foundation.py) but pointed at line_stops geometry
instead of foundation.build_usable_lines, plus anomaly.trip_features for
the per-trip aggregation (reused, not reimplemented -- same principle as
tier 6 of line_stops). ran on the SAME 30-day window as notebook 10
(d20260523-d20260621) for a fair before/after vs foundation_arrivals_
full.parquet.

BUG FOUND (real, pre-existing, not introduced this session): anomaly.
trip_features() computed match_rate=("matched","mean") AFTER already
filtering the frame to matched==True rows only (`matched = fa[fa[
"matched"]].copy()` ran BEFORE the groupby). mean of an all-True column
is always 1.0 by construction -- this feature has been mathematically
CONSTANT in the training data for the ALREADY-DEPLOYED anomaly model.
impact: dead weight in the Isolation Forest (zero-variance feature after
scaling), and the anomaly-explanation layer could never surface "mauvais
suivi GPS / hors itineraire" as a reason despite having that exact
message written in _REASON_BUILDERS. fixed by computing match_rate on
the FULL per-trip window (all stops, matched or not) as a separate
groupby, merged back onto the matched-only dwell/distance aggregates
(those correctly stay matched-only, same principle as the dist_m p90 bug
from notebook 10). verified: match_rate now shows real variance (min
0.136, mean 0.807, max 1.0 on a spot check) instead of a flat 1.0.

result vs foundation_arrivals_full.parquet, same 30-day window:
                          old (parquet)   new (line_stops)   delta
  trips                   426             915                +489
  full trips              299 (70.2%)     272 (29.7%)         -27 *see caveat
  stop-visit rows         3,901           7,951              +4,050
  matched stop-visits     2,297 (58.9%)   5,970 (75.1%)       +13.4pts
  distinct lines w/trips  11              26                  +15

*full-trip caveat, traced precisely, NOT a quality regression: of the 11
lines present in the old parquet, 10 have virtually identical geometry
old vs new (stop count + route length match to within rounding --
confirms these resolve via the same primary source in both systems, no
change). excluding the 11th (TCV line 3), those 10 lines actually show a
HIGHER full-trip rate in the new system: 62.7% vs 43.8%. the entire
aggregate drop is driven by TCV/3 alone: old geometry was 6 stops/5.8km
(TCV was one of the sparsest-covered companies at the very start of this
whole effort), new geometry is 15 stops/14.3km -- richer and confirmed

====================================================================
GPS PING CHARACTERISTICS -- what the BARGOU/bus-990 trace actually showed
====================================================================
(worth its own section -- this is a real, generalizable characteristic
of the GPS data itself, not specific to the triangulation investigation
it came from, and it's the most likely TRUE explanation for why
triangulation keeps failing across every company tried, not just
S.R.T.SELIANA)

concrete numbers from one worked example (bus 990, S.R.T.SELIANA,
2026/01/27):
- 1,644 pings that day, median gap between consecutive pings = 5.0s
  (excellent density -- consistent with the ~5-13s figures found
  elsewhere in this project for actively-tracked buses)
- but: 2 gaps that day exceeded 2 minutes, the longest was 9,485s
  (158 minutes -- over 2.5 hours of total signal loss in ONE gap)
- total dark time that day: 178 minutes out of the day's tracked span
- the specific ticket used for triangulation (07:45:00) landed inside
  one of these gaps -- nearest real ping was 07:38:34, 6.5 minutes
  before the ticket time, outside the +/-3min matching window

pattern: GPS tracking quality is NOT uniformly good-or-bad per bus/
company -- it's "swiss cheese." the SAME bus, same day, same company,
can have near-perfect 5-second density for hours and then vanish for
over 2.5 hours with no warning or pattern. average/median ping density
looks great in aggregate (which is why nothing in the companies table's
has_gps / gps_days_sampled signals would have predicted this failure --
those measure PRESENCE of tracking that day, not CONTINUITY within the
day). a random ticket timestamp has a real, seemingly unpredictable
chance of landing in one of these gaps regardless of how good the
company's overall GPS coverage looks.

implication for triangulation specifically: with a +/-3min matching
window and dark periods routinely running 30min-2.5h+, a fair number of
individual ticket-to-ping match attempts are structurally doomed before
the algorithm even runs, no matter how good the year-sampling/
disambiguation logic is. this is very likely the dominant reason 0/34
happened even on S.R.T.SELIANA/S.R.T.K -- companies whose ticket-pattern
(linear, non-shuttle) and bus-code linkage both checked out fine. WIDENING
the matching window (e.g. +/-3min -> +/-10-15min) would reduce this
specific failure mode somewhat, at the cost of looser position precision
(a bus moving between stops for 10-15 min covers real distance, so a
wider window trades timestamp-matching success against spatial accuracy
of the resulting coordinate) -- not tried this session, a reasonable
next experiment if triangulation is revisited later.

separately: ticket timestamps only have MINUTE precision (no seconds),
confirmed by the same bus/day showing 78 tickets across only 31 distinct
timestamps, with up to 12 tickets sharing one exact minute. this is
normal (several passengers boarding within the same minute naturally
share a stamp) and NOT itself a bug or the cause of the triangulation
failures -- the +/-3min window already comfortably exceeds 1-minute
timestamp granularity, so this doesn't materially compound the dark-
period problem above.
